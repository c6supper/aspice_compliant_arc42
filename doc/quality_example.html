Examples of quality requirements for software

About quality

The quality of a product or system is generally defined as a set of properties or features .

In practice, some categories (generic terms) have been established for frequently occurring quality requirements (synonym: quality goals), essentially shaped by the conceptual models of DIN / ISO 9126 and 25010.

Figure 1. ISO 25010 Quality Model

The categories

  • Changeability

  • Usability

  • Efficiency

  • reliability

  • Operability

  • Other (including functionality)

Classification can be ambiguous

Some quality goals or requirements belong to several "generic terms" or categories - it was therefore sometimes difficult for us to assign the examples to categories or chapters.

If you cannot find your specific quality requirements in the breakdown, please have a look in the appendix: Q characteristics - there you will find an extensive table of the Q characteristics with their synonyms and generic terms.

Formulation of scenarios

A (quality) scenario describes the reaction (1) of a system (3) when an event or stimulus (2) occurs.

(1) reaction of a system

The reaction can relate to software, hardware or a database, or to persons, roles or organizations involved.

The reaction of the system and thus the resulting behavior should always be formulated in scenarios that can be measured or decided .

(2) event or stimulus

Differentiation between usage event and change event.

Usage event

A user triggers a function while working with the system, for example:

  • selects a menu item / function

  • clicks a button on a graphical user interface

  • starts processing

Change event

A stakeholder changes something in the system, for example:

  • modifies source code or configuration of the system

  • changes to the hardware of the system

  • changes the deployment of the system

A stakeholder changes something in the processes or organizations involved.

(3) system

The term system is very broad here: This includes software, software components, hardware, networks, middleware, databases, but also all the people, roles or organizations involved.

1. Maintainability

1.1. What does maintainability mean?

Maintainability

Effort that is necessary to carry out specified changes. Changes are corrections, improvements or adjustments to changes in the environment, requirements and functional specifications.

According to DIN / ISO 25010, the following partial features are part of the changeability:

Modularity:

Expenditure for the creation and maintenance of a good functional and technical breakdown of the application into building blocks

Reusability:

The effort involved in using blocks in a context other than their original context

Analyzability:

Effort to diagnose defects or causes of failure or to determine parts in need of change.

Modifiability:

Effort to carry out improvements, to correct errors or to adapt to changes in the environment

Change stability:

Likelihood of unexpected effects of changes occurring.

Testability:

The effort required to check the changed software

Colloquially, changeability often means flexibility, adaptability, maintainability or expandability.

1.2. Maintainability scenarios

1.2.1. Scenario: Updating the operating system on end devices

Operating system updates (especially security patches) must be able to be fully distributed in less than 4 weeks on the (more than 1000) end devices distributed in the field.

Business goals

Operational readiness
trigger

Operating system updates are available
reaction

The updates are installed on all end devices.
Target value

4 weeks

1.2.2. Scenario: New connector

If the data import component is to read in data from an additional data source, the necessary connector must be developed, integrated and put into operation within three person-months.

Business goals

Flexibility, responsiveness, courtesy
trigger

It is necessary to import data from an additional data source
reaction

The new connector is programmed and operational.
Target value

3 person-months

1.2.3. Scenario: Annual update

The data output format of the annual booking report is adjusted every year on December 31st in order to comply with legal and tax changes. The system must be able to generate the new format.

Business goals

Operational readiness
trigger

Provision of the new specification of the data output format for booking reports
reaction

The booking and reporting components have been changed and deliver the booking report in accordance with the new specification.
Target value

The change must be implemented in less than 80 person hours.

1.2.4. Scenario: develop plugins

An independent developer can create plug-ins for the system.

Business goals

Allow the "community" to expand the basic functionality of the system
trigger

Independent developer (a "third party") would like to create its own extension ("plug-in") for the system
reaction

Independent developers are able to expand the system on the basis of predefined expansion points. In addition to the system, only open source components are required for this.
Target value

Simple plug-ins should be implementable within 8 person hours.

1.2.5. Scenario: changing the algorithm

Change of the route calculation algorithm.

Business goals

Implement changes or improvements in route optimization as quickly and easily as possible
trigger

The developer would like to develop an alternative algorithm for calculating suitable routes and integrate it into the overall system.
reaction

The route calculation algorithm has been replaced. There are no side effects on other components of the system.
Target value

Only a single software component (module, component, package) is affected; the actual exchange of the existing algorithm for an alternative algorithm takes place within 4 hours (including any adjustments to the build environment).

1.2.6. Scenario: New field in master data

A developer wants to add a new field to the XY master data.

Business goals

Ability to change / adapt the domain model (technical data structure) including user interface and database.
trigger

The specialist, business or marketing department needs a new attribute of the type "String" (maximum length 80 characters) in the XY master data
reaction

A developer identifies the associated program parts, makes the necessary extensions, tests the extensions in the integration and system test environment.
Target value

All necessary changes and associated tests are completed within 40 hours (effort).

1.2.7. Scenario: Supports ANSI SQL-92 compatible database systems.

Offer the system to customers with different DB systems.

Business objective (s)

Flexibility, the system offers customers with different database systems.
trigger

A customer would like to use the system with a (previously unsupported) (ANSI-SQL-92 compatible) database.
reaction

Developers and testers commission the system with the database and carry out all system tests, especially the SQL-92 compatibility tests.
Target value

If the database successfully passes the compatibility tests, all other system tests are also successful. In this case:

  • the workload for system setup, commissioning and test execution a maximum of 20 hours

  • the total duration is a maximum of one calendar week.

1.2.8. Scenario: understandability of generated code

Generated code for test cases is easy to understand and can be modified manually.

Quality goals

Comprehensibility, testability, modifiability
Business objective (s)

Testers can easily understand the automatically generated code for test cases and adapt it to their own needs.
trigger

A tester describes a test case in a (semi-formal) language.
reaction

Test code (in C, Java or C #) for certain parts of the system is automatically generated by the code generator for test cases on the basis of this test case description.
Target value

A tester can understand the generated (C or Java) source code for a test case on average in less than 30 minutes and make simple changes to it.
annotation

In the scenario mentioned here, testers change generated code. Depending on the type of code generator, this changed code could be overwritten when it is regenerated. Corresponding requirements would have to be made in other scenarios.

1.2.9. Scenario: Quick localization of errors

Developers can localize the cause of failed test cases in a short time.

Business goals

Short troubleshooting times, easy modifiability, low risk of changes
trigger

Developers, testers or the build system execute automatic test cases (unit or integration tests). A test fails.
reaction

From the error and / or log messages, a developer can localize the cause / trigger of the error in a short time.
Target value

Developers find the cause in less than 10 minutes on average.

1.2.10. Scenario: New function

The client requests a new function in the system. Developers quickly identify the right place within the source code for this extension.

Business objective (s)

Fast cost assessment of customer requirements
trigger

Change request
reaction

Developers look for the appropriate place within the source code for the desired extension. Building on this, he can better evaluate / estimate the effort and risks of the desired expansion.
Target value

Developers can find the right job in less than two hours.

1.2.11. Scenario: Short implementation time for a new function

A new function of medium complexity should be able to be implemented in a short time.

Business goals

A new function of medium complexity should be able to be implemented in a short time.
trigger

Change request
reaction

Developers identify the modules to be adapted, implement the required function and automated test cases (unit tests) in them
Target value

The effort to complete the new function is less than 5 days.

1.2.12. Scenario: Extended log messages

The previous format of the log messages is no longer sufficient for the operator. All log messages must be supplemented with additional information.

Business objective (s)

Flexible adaptation to changed / extended logging requirements.
trigger

The operator needs additional information in log messages (e.g. IP address of the executing web server, session ID, etc.).
reaction

Developers must identify, adapt, test and commission the affected areas in the source code.
Target value

The implementation takes place within less than 40 hours.
annotation

For this scenario, the relationship between the target value and the scope of the affected source code (LoC) is relevant.

1.2.13. Scenario: Implement the report in a short time

A developer wants to implement a report on all bookings for a day

Business objective (s)

A new function (of low complexity) should be able to be implemented and tested in a short time
trigger

Client needs the results of the booking report
reaction

  • Developer implements this function within 3 working days.

  • Tester tests this function against the specification within 2 working days.
Target value

The new report is available with a maximum effort of 16 hours and after a maximum duration of 5 working days.

1.2.14. Scenario: Add authentication to the interface

Developer extends the external XY interface to include authentication.

Business objective (s)

The XY interface now requires secure authentication.
trigger

Regulations or customer requirements require secure authentication via OAuth 2.0 of the XY interface
reaction

The developer is expanding the interface to include OAuth 2.0 based authentication.
Target value

Securing the XY interface requires a development effort of a maximum of 5 PT.

1.2.15. Scenario: Replace commercial database with open source database

It must be possible to replace the commercial database management system used by a license-free, open source product.

Business goals

If necessary and in certain application scenarios of the system, license costs must be reduced.
trigger

The operator / customer of the system would like to reduce the license costs of the database system used.
reaction

Developers are replacing the standard commercial database with an open source, license-free DBMS. It is ensured that all functional requirements are met. This is demonstrated by the execution of the automated integration, system and acceptance tests.
Target value

  • The additional support of the alternative database management system can be carried out with less than 40 days of effort.

  • The runtimes of the 15 most important use cases deteriorate by a maximum of 15% compared to the commercial database.

  • The runtimes of all other use cases deteriorate by a maximum of 25% compared to the commercial bank.

1.2.16. Scenario: Expand business process

The business process XY can be supplemented with additional processing steps at runtime

Business goals

Dynamic expansion of the business process XY contributes to the satisfaction of the users.
trigger

To adapt to market needs, a developer or architect expands the business process XY by an additional step, while the active process instances of XY are processed by the system.
context

More than 20 users have saved incomplete projects (data) based on the current XY process.
reaction

The system updates the existing data of the incomplete projects independently and without data loss (automatic migration of user-specific data to the new version of XY).
Target value

Changing the XY process does not take more than 80 person hours (effort).

1.2.17. Scenario: Change to the business process

The individual processing step AB within use case XY is declared invalid by the regulatory authority and removed from the system. The data processed by the system are not affected.

Business objective (s)

The change to use case XY can be carried out at low cost and without negative effects.
trigger

The legislature, represented by the regulatory authority, prohibits the use of processing step AB.
reaction

A developer or architect removes the processing step AB in the system (by deleting the corresponding calls or by reconfiguring the process flows).

This change has no effect on the user / customer data available in the system with regard to the XY application. An (automatic) migration of some data is permitted, but must not exceed the 24 hour limit.
Target value

The change requires a maximum of 24 hours with a maximum of 48 person hours. After this time, the system is fully functional again.

1.2.18. Scenario: Extension by an automated test suite

Extend the X subsystem with a fully automated test suite.

Business objective (s)

Improve the changeability and testability of the system.
trigger

Customer announces extensive change requests to the X subsystem. The better changeability and testability reduce the risk of these changes.
reaction

Developers implement unit and integration tests for all classes and interfaces of the X subsystem.
Target value

The total effort of the changes is less than 200 man-days.

1.2.19. Scenario: New type of client

Expand the XY system with a mobile client (Android, iOS) without affecting the performance of the other GUI parts.

Business goals

The introduction of new access routes broadens the possible customer base and increases the attractiveness of the system.
trigger

Customers and consumers expect (as a matter of course) native mobile clients as part of the product portfolio.
reaction

Development team designs and implements mobile clients for the XY system as well as the required (data) interface.
Target value

  • The previous (browser and rich) clients are not impaired in their function and performance in any way.

  • With 100 parallel browser users and 100 mobile clients logged on at the same time, a maximum of 3% of the data accesses may take a maximum of 20% more time than before the introduction of the mobile clients.
annotation

This scenario describes several quality features: attractiveness, efficiency / performance, availability, robustness and changeability.

1.2.20. Scenario: Independent expansion of a subsystem

Extensions or changes to a subsystem should be possible independently of all other subsystems.

Business objective (s)

Easily changeable, quick reaction to errors.
trigger

Development team change within a subsystem - the external interfaces of this subsystem remain identical!
Target value

No other subsystem needs to be changed. The following applies to all other subsystems:

  • The source code remains the same

  • Compile, build and test processes remain identical

  • Deployment, installation and configuration remain the same
annotation

This requirement means that all subsystems only communicate via their public ("official") interfaces - and that no subsystem uses or exploits any other's internals. A particular challenge when it comes to coupling via data or data structures.

2. Usability

2.1. What does usability mean?

Effort required for use and individual assessment of use by a specified or required user group. This also includes software ergonomics .

According to DIN / ISO 25010, suitability for use includes the following partial characteristics:

Comprehensibility:

Effort for the user to understand the concept and application.

Learnability:

Effort for the user to learn the application (e.g. operation, input, output)

Usability:

Effort for the user to operate the application.

Usefulness:

.

User interface aesthetics:

.

2.2. Usability scenarios

2.2.1. Scenario: Simple usability of test tools

The interactive test tool must be easy to use. Testers who carry out the user acceptance tests (BA) must be able to learn how to operate the test tool quickly.

Quality target

Learnability
Business objective (s)

Productive and quick acceptance tests, quick runs
trigger

A new BA tester should use the test tool to test a software release.
reaction

The tester learns the test tool and can then operate the test tool completely.
Target value

A user acceptance tester needs less than two hours of training to learn the test tool.

2.2.2. Scenario: Consistent keyboard shortcuts

End users / end users can use identical keyboard shortcuts in all independently developed modules of the system.

Quality target

Learnability
Business objective (s)

Consistent user experience throughout the product
trigger

User wants to use keyboard to navigate within the system
reaction

User can perform the same functions as when using the mouse.
Target value

The keyboard shortcuts are assigned uniformly. When using the keyboard, all functions can be used faster or at least as quickly as when using the mouse.

2.2.3. Scenario: Rapid comprehension of information

Users should also be able to quickly grasp information that is not related to the current screen mask.

Quality goals

Usability, efficiency
Business objective (s)

Easy to use, easy to navigate within the system
trigger

The user works with the system. While working on a screen mask, he would like to enter or edit information on a completely different topic.
reaction

The user can easily navigate to the desired topic and simply return to the current topic after editing it.
Target value

You can navigate to the desired topic in less than 10 seconds, and return to the current topic with just one push of a button / mouse click.

2.2.4. Scenario: Indication of incorrect entries

Users are always advised of inconsistent or incorrect entries.

Quality target

Usability, fault tolerance
Business objective (s)

Consistency in user input
trigger

User enters data.
reaction

The system accepts correct input data and rejects inconsistent or incorrect input data.
Target value

In the event of inconsistent or incorrect input data, the system issues a suitable message that clearly and simply shows the error or inconsistency.

2.2.5. Scenario: User-friendly behavior in the backend process

If a user interrupts the PDF generation of the XY report, the system stops this generation and returns control to the user interface within 15 seconds.

Quality target

Usability
Business objective (s)

Improve the usability of the pdf generation
trigger

The user would like to interrupt the PDF generation of the XY report (for example due to previous incorrect entries or other reasons) and clicks the "Cancel" button
reaction

The system interrupts the generation, saves the previous generation status (for possible continuations) and transfers control to the user interface.
Target value

The user is given control of the UI within 15 seconds (ie within 15 seconds at the latest, all the generation processes involved have successfully acknowledged the cancel command).
comment

The generation of this report runs in several parallel threads (or processes), possibly even on several different processors or (virtual) machines. The interruption must occur synchronously and consistently across all of these execution instances.

2.2.6. Scenario: Meaningful error messages without a crash

If an error situation occurs, this is indicated to the user in meaningful messages. The system does not crash in exceptional situations (memory overflow, hardware error), but only shuts down in a controlled manner.

Quality goals

Error tolerance, usability
Business objective (s)

Improve usability (and perceived reliability). Allow users to correct incorrect combinations of input data without crashing the system.
trigger

An error / exceptional situation occurs in the infrastructure (memory overflow, out-of-memory, hardware error).
reaction

The system detects the error, reports (as far as possible) to the user and shuts down in a controlled manner.
Target value

Error detection occurs within 15 seconds, notification to the user (if still possible) within 1 second, shutdown within 15 seconds.

2.2.7. Scenario: Status of long-running processes recognizable

The system shows the progress of the long running XY conversion processes in the graphical user interface.

Quality goals

Usability
Business objective (s)

Progress monitoring, ease of use
trigger

User would like to be informed about the progress of long running calculations / processes.
reaction

The system shows the number of data records processed so far, the processed data volume in megabytes and the estimated remaining workload on the GUI.
Target value

This information is updated at least every 60 seconds - at most every 5 seconds.
annotation

This XY conversion takes 1-18 hours.

2.2.8. Scenario: Compliance with the operating system-specific user experience guidelines

The system should meet the requirements and suggestions of the Microsoft Windows User Experience Guidelines for Windows 8.

Quality target

Aesthetics of the user interface, learnability
Business objective (s)

Uniform appearance and behavior corresponding to the Windows 8 look and feel.
trigger

The user interface and interactive components of the system are to be redesigned and implemented.
Target value

A competent auditor tests compliance with the above guidelines without restriction.

2.2.9. Scenario: User interactions on a graphical user interface

All user interaction and messages in the system are implemented as a GUI.

Quality target

usefulness
Business objective (s)

Increase the acceptance and productivity of working with the system.
trigger

All system requirements.
Target value

The system does not require any user interaction outside of the graphical user interface.

3. Efficiency ( performance efficiency )

3.1. What does efficiency mean?

According to DIN / ISO 9126: Relationship between the level of performance of the software and the scope of the equipment used under specified conditions.

In practice, often simply referred to as performance , processing speed, response time, scalability, throughput, storage requirements or quantity structure.

According to DIN / ISO 9126, efficiency includes the following partial characteristics:

  • Time behavior : response and processing times as well as throughput during function execution.

  • Consumption behavior : number, quantity and duration of the resources required to fulfill the functions.

Note: You can objectively measure the efficiency of existing software "on the living object". In this respect, it is relatively easy to check whether the efficiency requirements for software are being met.

3.2. Scenarios for efficiency

3.2.1. Scenario: Rapid generation of test data

Rapid generation of large amounts of test data for the XY system.

Business objective (s)

Effective testing, testing with large amounts of data.
trigger

A tester needs test data to test the XY system.
reaction

The test data generator generates 1 gigabyte of suitable, technically correct test data.
Target value

Running time less than an hour.

3.2.2. Scenario: Diagnosis has little effect on execution time

The diagnosis subsystem has only a minor influence on the execution time of functions and transactions in the system.

Business objective (s)

Accurate reporting on runtimes and precise error diagnosis without affecting runtimes.
trigger

The user, tester or administrator calls a diagnostic function.
reaction

The system continues to work without restriction.
Target value

All functions and transactions of the system are functionally correct. Runtimes are no more than 5% higher than when the diagnostic subsystem is switched off.

3.2.3. Scenario: Generation of reports in less than 4 hours

Generation of all reports and lists required for the month-end closing within 4 hours.

Business objective (s)

High-performance and punctual monthly closing
trigger

The controlling or finance department starts the month-end closing after the booking deadline.
reaction

The system generates all necessary reports and lists.
Target value

The generation ends after a runtime of 4 hours at the latest, the first (simple) reports are available for controlling after a runtime of 30 minutes.

3.2.4. Scenario: Integration tests in less than 15 minutes

All integration tests of the XY subsystem can be carried out automatically within 15 minutes.

Business objective (s)

Low-risk changes and enhancements.
trigger

The developer makes a change to the source code in subsystem XY and then starts the automated test suite (the integration and unit tests) of this subsystem.
reaction

The test framework executes all test cases and reports the test results to the user.
Target value

All test cases are completely executed in less than 15 minutes.
comment

Load, performance or stress tests can also run longer, independent of the integration tests. These are not part of this scenario.

3.2.5. Scenario: data within 3 seconds

In 90% of all cases, users will get the XY data within 3 seconds.

Business objective (s)

Several users can work in parallel with an acceptable runtime
trigger

10 users working in parallel request the XY data from the system.
reaction

The system displays the correct data for all requesting users.
Target value

With at least 9 out of 10 of these users, this request takes 3 seconds or less (in 90% of the requests for the XY data, the system responds in 3 seconds or faster).

3.2.6. Scenario: high performance with 200 simultaneous users

With 200 or more users logged on at the same time, the system still performs well.

Business objective (s)

Ability of the system to maintain its functionality, especially at the GUI, even with several parallel users.
trigger

  • 200 or more users are logged on to the system.

  • 20 or more users work simultaneously in the XY dialog or start a YZ calculation.
reaction

The system works normally and serves all users.
Target value

The system's reactions in the XY dialog take place within 2 seconds. The system will perform the YZ calculation in less than 5 seconds. Note: scalability

3.2.7. Scenario: Comprehensive reports in less than 1 second

The XY system is also supposed to save extensive user-defined reports in less than 1 second.

Business objective (s)

Reactive user interface increases ease of use.

Trigger: The user has configured a specific report and saves it using the "save" function.

Response and target value: The system saves the entire report definition (in xml format) in less than 1 second.

3.2.8. Scenario: The configuration interface appears in less than 2 seconds

The graphical configuration interface adapted to the user profile appears in less than 2 seconds.

Business objective (s)

Reactive user interface increases ease of use.
trigger

User starts the configuration interface for reports.
Response and target value

When the report configuration interface is started, a graphic interface prepared according to the user profile appears in less than 2 seconds.

3.2.9. Scenario: Use of a maximum of 1GB RAM

The application should only require 1 GB of RAM as a base and no more than another 5 MB per user logged in at the same time

Business objective (s)

The application is intended to efficiently use the system resources made available and to enable the growth of employees.
trigger

A user logs on to the system.
Response and target value

The application should only require 1 GB of RAM as a base and no more than another 5 MB per user logged in at the same time. After logging out, the 5 MB must be freely available again.

4. Reliability

4.1. What does reliability mean?

Ability of the software to maintain its level of performance under specified conditions for a specified period of time.

According to DIN / ISO 9126, reliability includes the following partial characteristics:

Maturity:

Low frequency of failure due to faulty conditions.

Fault tolerance:

Ability to maintain a specified level of performance in the event of software errors or non-compliance with its specified interface.

Recoverability:

Ability to restore performance levels in the event of failure and recover the data directly affected.

4.2. Scenarios for reliability

4.2.1. Scenario: Detailed information about errors

If the data import fails, the system provides detailed information about the error (s) that have occurred.

Business objective (s)

Ability to quickly identify, locate and correct errors in data import.
trigger

Data import fails.
reaction

The system collects the information relevant to error diagnosis and rectification (type of error, affected data records, time, last successful action carried out, etc.)
Target value

Relevant information is written to the log file less than 30 seconds after the error occurs and sent by smtp mail to <[ x@y.com ] (mailto: x@y.com)>.

4.2.2. Scenario: measurement accuracy for time measurements

The diagnosis subsystem should have the same measuring accuracy for time measurements as corresponding external tools.

Business objective (s)

External measuring tools for the execution times of transactions provide identical results to the internal diagnostic subsystem.
trigger

A user initiates any transaction in the system. The internal diagnosis is configured to "on".
reaction

The internal diagnostic subsystem stores the start and end time of the transaction.
Target value

The times measured by the internal diagnostic subsystem correspond to values ​​within 5% that external tools have determined for the same transaction.

4.2.3. Scenario: Failover for containers

The system has a failover for the servlet container.

Business objective (s)

Problem-free handling of general error and failure situations
trigger

The servlet container crashes due to a software problem - the hardware and operating system are still available.
Response and target value

The monitoring system detects the failure within 1 second, provides a replacement container within 15 seconds and is fully functional again after 120 seconds at the latest.
Remarks

For a specific system, it would also have to be specified whether and to what extent the currently active sessions of the failed container must be backed up and transferred to the replacement system.

4.2.4. Scenario: No crashes if there is insufficient memory

The system processes data in the range of up to several gigabytes during the PDF generation and file conversion (in memory). If there is a shortage or overflow of memory, the system must not crash, but must write meaningful log messages, terminate the generation in a controlled manner and notify the user.

Business objective (s)

Reliability of the system even with extensive and voluminous generation and conversion tasks.
trigger

The system generates or converts data, possibly distributed over several threads, processes or nodes. One or more of these threads, processes, or nodes is experiencing a memory overflow.
reaction

The system finishes the relevant processes in a controlled manner and saves the intermediate status of the generation / conversion for later use. It generates a suitable log message and informs the respective user about the situation.
Target value

Memory overflow is recognized within 15 seconds, and all processes involved are terminated in a controlled manner within a further 15 seconds.
comment

also usability

4.2.5. Scenario: Functional correctness even with undervoltage

The system behaves functionally correct in all respects even in the event of undervoltage in the hardware sensors (up to 15% below the nominal voltage).

Business objective (s)

reliability
trigger

The voltage of the hardware sensors (e.g. sensor for measuring the paper transport speed, the flow rate, etc.) falls at most 15% below the prescribed nominal voltage.
reaction

All system functions continue to work correctly.
comment

Some sensor types behave uncontrollably when there is undervoltage, some work more slowly, others imprecisely, others not at all. The system functions must therefore monitor the voltage of the sensors and react accordingly to voltage problems.

4.2.6. Scenario: No data loss in the event of voltage loss or undervoltage.

Business objective (s)

Robustness against fluctuations or failures in the electrical supply. The system does not lose any data in the event of a power loss or undervoltage (of the entire mains supply).
trigger

The supply voltage fails or fluctuates by up to 25%.
reaction

The system is protected by a redundant power supply long enough to be able to consistently save all data in the memory on long-term storage media in the event of a loss of the regular power supply.
Target value

Loss of voltage or undervoltage is recognized within 200msec.

4.2.7. Scenario: The system offers a failover of the servlet container.

Business objective (s)

The system should handle failures of the general infrastructure (especially servlet containers) without any problems and without a crash.

trigger

The servlet container (necessary for many system functions) fails.

reaction

The system detects the failure and transfers all data / sessions that are still available to a replacement servlet container.

Target value

  • Discover the error in the servlet container within 1 second.

  • Starts up the hot / cold standby servlet container within 30 seconds.

  • After 180 seconds, the system has restored all functionality from before the failure.

4.2.8. Scenario: stability even with continuous use

Even in continuous operation, the system is reasonably stable and robust to online users.

Business objective (s)

Stability, user comfort
trigger

End users will use the system for at least 8 hours without restarting, logging out, or other interruption. At least 20 parallel users are permanently logged on to the system - a maximum of 1000 parallel users.
reaction

The system works correctly for all registered users. This also applies to users who have been logged in for a long time.
Target value

During the entire time, the users do not experience a crash. Note: It must at least appear to the user that the system is behaving in a stable manner. The system must be able to conceal server-side problems from the users or compensate them with standby systems or failover.

5. Operability

5.1. What does operability mean?

Operability is not defined in DIN / ISO 9126.

In practice, this includes the following sub-characteristics:

Analyzability:

Effort to diagnose defects or causes of failure or to determine parts in need of change.

Installability:

The effort required to install the software in a defined environment.

Transferability:

Suitability of the software to be transferred from one environment to another. Environment can include organizational, hardware or software environment. Partly referred to as "portability".

Interchangeability:

Possibility of using this software instead of a specified other in the environment of that software, as well as the effort required for this.

Coexistence:

Ability of the software to work alongside another with similar or identical functions

5.2. Operability scenarios

5.2.1. Scenario:

The tool for measuring the current database performance must run under MySQL, Oracle and DB2.

Business objective (s)
response

The tool must be able to establish a connection to all of the databases mentioned and be able to display the respective DB structure.
Target value

The connected DB type is initialized and recognized in less than 30 seconds.

5.2.2. Scenario: automatic installation

The XY system, including all of the required software components, can be installed fully automatically.

Business objective (s)

Short release cycles in order to be able to put quick expansions or bug fixes productively
trigger

Operator (or administrator) would like to install a new version of XY (within a new virtual machine or new hardware)
reaction

An automatic installer (script, program) installs all parts of the XY system as well as all required additional software components (such as database, middleware, crypto modules) on the basis of an Ubuntu Linux standard installation. All required data (e.g. database initialization, LDAP standard user) are also created.
Target value

The entire installation takes a maximum of 30 minutes based on an already installed Ubuntu Linux.

5.2.3. Scenario: longevity

The software can be ported to the latest hardware and software (operating systems, DB systems, etc.) over a period of 20 years, for which no more than 5 man-days per month have to be invested.

5.2.4. Scenario: Installation without internet access

Independence of the installation from coincidences of the Internet

The software can be installed on a system without internet access.

Suitability of the functions for specified tasks, eg task-oriented combination of functions from sub-functions.

Ability to interact with given systems. This also includes embedding in the operating infrastructure.

Fulfillment of application-specific standards, agreements, legal provisions and similar regulations.

6.2. Other scenarios

6.2.1. Scenario: availability for different database systems

The system (tool) is used to measure the performance of database systems. It must be available for different database systems.

Business goal

Portability, efficiency, operability. Versatile tool, should be available for different database systems
trigger

The tool is connected (via dialog, command line or programmatically) to a database system: Oracle, DB2, Sybase, MySQL, PostgreSQL
reaction

The tool successfully establishes the connection to the respective DBMS - the DB structure is visible in the tool.
Target value

The "connect" to the database takes place within 30 seconds.

6.2.2. Scenario: Core function portable on iOS and Android

The core functions of the Mac OS software can be reused on iOS and Android.

Quality goals

Reusability, interchangeability
Business goal

Fast time-to-market, reduced development costs, market expansion
trigger

Code is designed for Mac OS
reaction

As far as possible (as far as the technology allows), the code can be reused on both iOS and Android.
Target value

The development team should reuse the core functions without having to implement or redesign them from scratch.

The exception to this are direct calls to the respective operating system functions.

6.2.3. Scenario: 60% test coverage for unit tests.

Quality goals

Testability, changeability
Business goal

Easy to expand and change thanks to high test coverage, quick feedback on possible side effects in the event of code changes.
trigger

Developer develops or changes a function / method / class.
reaction

The affected function / method / class is checked by unit tests.
Target value

The unit tests achieve over 60% path coverage.

6.2.4. Scenario: Simply adding new tests.

Testers should be able to add new tests to existing test suites with little effort.

Quality goals

Testability, changeability
Business goal

Ability to easily integrate new tests into existing test suites.
trigger

Tester wants to add a new test to an existing test suite
reaction

The test is added without modifying the code of the system itself (ie the productive code).
Target value

There is no need to recompile or reconfigure the system (more precisely: the productive code).
annotation

However, test code or test configurations may be changed.

6.2.5. Scenario: Similar results for stochastic test series

If two series of stochastic tests are carried out, the results are 90% similar.

Quality goals

Testability, consistency, traceability. Consistent test results even with stochastic tests or test series.
trigger

Test scenario or test series with a proportion of randomly determined test data is carried out
reaction

Similar test results.
Target value

The results of two test series are similar in 90% of all individual tests.

6.2.6. Scenario: test execution with a single command

A tester wants to be able to run multiple test scenarios with a single command.

Business goal

Efficiency, testability
trigger

The test administrator or tester wants to run multiple tests or test scenarios
reaction

Aggregation or execution of several tests one after the other.
Target value

Running multiple tests requires a single command from the tester or test administrator.

6.2.7. Scenario: X data accessible to external systems

The X data of the Y system should be accessible for an external tool for requirements analysis and management (e.g. Requisite-Pro, Enterprise-Architect, etc.).

Quality target

Interoperability
Business objective (s)

Data compatibility with standard tools for requirements analysis is an advantage over competitors.
trigger

An end user wants to transfer the X data of the Y system into one of the supported request tools.
context

20 users have entered X data in the form of individual projects in the system. Each of these projects contains at least one, at most 100 different requirements.
reaction

The Y system exports the relevant X data to the requirement tool (Requisite-Pro or Enterprise-Architect).
Target value

  • There are no errors when exporting the data.

  • At least 98% of the requirements contained in the Y system are exported correctly.

  • All non-exported requirements are reported to the affected users as errors.
